# 成行 SureGo MVP1.0 我能讲明白版

这份文档只解决一个问题：

> 我能不能用 3 分钟，把 MVP1.0 讲清楚，让自己和内部团队先对齐。

它不是完整 PRD，也不是接口文档。它是产品经理、UI、开发、运营先统一方向用的版本说明。

## 1. 产品一句话

成行 SureGo 是面向大学生的同校搭子小程序，帮助用户围绕具体活动、任务和长期目标，快速发现搭子、建立连接、发起活动，并把饭局、运动、自习、游戏、项目组队和长期搭子这些事真的约成。

更口语一点说：

> 想吃饭、运动、自习、游戏开黑、组队比赛或找长期搭子时，不用在群里乱喊。打开成行，就能看到同校同学正在组什么局、找什么搭子；合适就报名、提交意向、私聊、拉群，最后把它变成一次真实成行。

成行不是“刷人”的社交产品，而是 **活动/任务导向的搭子社交产品**。

用户不是先看一个人，再漫无目的地聊天；而是先看到一个明确的活动、任务或长期目标，再决定是否报名、提交意向、关注、私聊、进群或一起发起成行活动。

## 2. 用户主路径

MVP1.0 有两个核心入口：

1. **成行**：负责明确活动的报名、赴约、核销和活动沉淀。
2. **搭子**：负责时间未定、长期关系、项目组队和任务型搭子的意向连接。

### 路径一：参加一个公开成行活动

```text
成行页 -> 活动详情 -> 报名/提交申请 -> 审核/候补/确认占位 -> 报名成功 -> 赴约信息 -> 到场核销
```

这条路径解决：

- 用户想马上参加一个明确活动
- 活动有相对明确的时间、地点、人数和规则
- 报名后进入赴约准备
- 到场后完成核销，证明活动真实发生

### 路径二：搭子并建立连接

```text
搭子页 -> 搭子帖详情 -> 提交意向/申请认识/申请加入 -> 发帖人确认 -> 关注/私聊/群聊 -> 决定下一步
```

这条路径解决：

- 用户想找人一起做某件事，但时间地点还没完全确定
- 用户想找长期搭子、游戏搭子、自习搭子、项目队友或黑客松队友
- 双方需要先通过帖子判断是否合适，再建立连接

### 路径三：从搭子帖转成成行活动

```text
搭子帖 -> 多人提交意向 -> 发帖人确认时间地点 -> 转化发局 -> 选择公开或私密 -> 生成成行活动
```

转化时有两种结果：

- **公开活动**：进入成行页，和普通活动一样被更多同校用户发现、报名和沉淀。
- **私密活动**：不进入成行页，只在参与者和发起人的我的页/活动管理里可见。

这条路径是 SureGo 的关键闭环：

> 搭子不是停留在互相看看，而是要顺畅地建联，再顺畅地转成真实活动。

### 路径四：发布

底部中间入口用醒目的 **加号** 表示发布。

```text
+ -> 发布活动局 / 发布搭子帖
```

发布活动局：

```text
选择场景 -> 填写时间地点人数规则 -> 发布成功 -> 发局工作台管理
```

发布搭子帖：

```text
选择搭子帖类型 -> 选择兴趣/场景 -> 填写结构化需求 -> 发布成功 -> 发帖工作台管理
```

## 3. MVP1.0 只做什么

MVP1.0 是演示版，目标是让自己和内部团队看懂完整产品逻辑，不要求所有能力都真实接后端。

第一版只做同校范围。

MVP1.0 做：

1. 成行页
2. 毕业季约拍专题页
3. 搭子页
4. 发布加号入口
5. 发布活动局
6. 发布搭子帖
7. 活动详情页
8. 搭子帖详情页
9. 报名/申请页
10. 审核和候补表达
11. 确认占位表达
12. 报名成功页
13. 我的页
14. 我的活动/我的搭子帖管理
15. 发局工作台
16. 发帖工作台
17. 通知页
18. 学生认证基础展示
19. 关注
20. 私聊
21. 群聊/拉群入口
22. 黑客松限时推荐专区
23. 长期搭子、约个时间、项目组队三类搭子帖

MVP1.0 支持线上和线下场景。

线下场景包括：

- 饭局
- 运动
- 自习
- 桌游/剧本杀
- 线下游戏或娱乐活动
- 生活方式活动
- 校园节日或主题活动

线上场景包括：

- 游戏开黑
- 线上自习/打卡
- 黑客松远程组队
- 项目协作
- 长期任务型搭子

黑客松放在搭子页顶部作为限时推荐主题。

以后这个位置可以轮换多个主题，例如：

- 黑客松组队
- 毕业季搭子
- 海棠节活动搭子
- 暑期学习搭子
- 考研自习搭子

## 4. MVP1.0 不做什么

为了把第一版做清楚，MVP1.0 暂时不做：

1. 独立 App，只做小程序/小程序形态演示
2. 多学校，只做同校
3. 真实支付
4. 完整支付、退款、结算系统
5. 复杂推荐算法
6. 完整信用体系
7. 完整长期搭子自动匹配算法
8. 大规模运营后台
9. 多城市推广
10. 完整商业化体系

这里要讲清楚：

- 不是不做支付，而是 MVP1.0 先不接真实支付，只做确认占位、诚意金、门票等前端表达，后续再接后端和支付能力。
- 不是不做信用，而是 MVP1.0 先展示基础信任信息，不做复杂信用体系。
- 不是不做长期搭子，而是 MVP1.0 先做搭子帖、标签筛选、提交意向、关注、私聊、群聊和转化发局。
- 私聊和群聊可以进入产品设计，但第一版技术实现可以先做前端演示或轻量能力。

## 6. 每个页面一句话说明

成行页：
让用户先看到毕业季约拍等专题推荐，再通过标签、排序和活动信息流看到同校已经明确时间、地点、人数和规则的活动局，并快速报名参加。

搭子页：
让用户看到同校正在找什么搭子，包括约个时间、长期搭子、项目组队和限时主题推荐。

发布入口：
用醒目的加号承接发起行为，让用户选择发布活动局或发布搭子帖。

发布活动局页：
让用户发布一个时间地点较明确、可以报名和核销的成行活动。

发布搭子帖页：
让用户发布一个结构化搭子需求，选择约个时间、长期搭子或项目组队，并填写目标、时间、地点、频率、希望对方和连接规则。

活动详情页：
让用户判断这个活动是否适合自己，包括内容、时间地点、人数缺口、费用规则、发起人、私聊/群聊入口和报名门槛。

搭子帖详情页：
让用户判断这个搭子需求是否适合自己，并提交意向、申请认识、申请加入、关注、私聊或进群。

报名/申请页：
让用户提交上车信息、回答问题，并确认基础规则。

确认占位页：
用前端表达名额确认、诚意金或门票规则，但 MVP1.0 不接真实支付。

报名成功页：
告诉用户报名或占位已经完成，以及下一步去我的页查看赴约和活动管理信息。

我的页：
承接个人资料、认证、我报名的活动、我发起的活动、我的搭子帖、待审核、候补、待核销、私聊、群聊和关注。

发局工作台：
让活动发起人管理报名审核、候补、成员、通知、核销、退款/争议状态表达。

发帖工作台：
让搭子帖发布者管理意向申请、通过/拒绝、关注、私聊/群聊，以及把搭子帖转成公开或私密成行活动。

通知页：
让用户看到审核结果、候补补位、行前提醒、发帖意向、私聊消息、群聊消息、局长通知、占位和退款相关提醒。

私聊页：
让双方在提交意向或互相关注后进行一对一沟通，用来确认是否适合一起成行。

群聊页：
让一个活动局、项目组队或搭子帖意向群体进行多人沟通，用来推动成行或协作。

学生认证页：
表达同校身份验证的价值和基础认证方式，MVP1.0 先做展示和流程表达。

他人主页/搭子主页：
让用户判断对方是否靠谱、是否有共同目标，并进行关注、私聊或查看 TA 的活动/搭子帖。

黑客松主题专区：
作为搭子页顶部限时推荐主题，集中展示项目方向、缺口角色、比赛节奏和申请加入入口。

## 最后一句话

MVP1.0 的判断标准是：

> 一个同校大学生能不能围绕一个具体活动、任务或长期目标，快速发现搭子需求，建立连接，并顺畅地转化成一次真实成行。

如果这条逻辑跑通，再继续做真实支付、完整后端、复杂信用、推荐算法、多校扩展和更完整的运营后台。
