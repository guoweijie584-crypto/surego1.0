# 项目鲁棒性缺陷全量排查报告

审计日期：2026-05-06  
审计范围：`D:\SureGo3\surego1.0` 当前项目全量源码、前端页面、前端 API 封装、uniCloud 云函数、数据库 schema、运行配置和依赖锁文件。  
报告边界：本报告仅记录鲁棒性、稳定性、容错性、并发一致性和兼容性问题，不记录安全风险和业务规则矛盾。

## 一、总体结论

当前项目具备基本的页面和云函数链路，但鲁棒性短板明显：多个页面的关键异步动作缺少 `try/catch/finally`，失败后会卡住 loading 或提交状态；云函数普遍缺少 schema 级参数校验和事务；报名、订单、签到等核心数据存在并发重复写和计数不一致风险；字段命名同时存在 camelCase 与 snake_case，导致查询结果遗漏；示例代码与生产代码混杂，增加多环境不可控因素。

## 二、鲁棒性问题清单

### R-01 报名提交失败后页面会永久停留在提交中

问题精准位置：`pages/activity/register.vue:246`、`pages/activity/register.vue:250`、`pages/activity/register.vue:265`

触发条件与复现步骤：进入活动报名页，填写报名信息后点击提交；断网、云函数抛错、token 过期或 `submitApplication` 返回异常；观察按钮和页面状态。

系统失效表现：`isSubmitting.value = true` 后没有 `try/catch/finally`，异常会中断函数，`isSubmitting` 无法恢复为 `false`；用户无法再次提交，也看不到明确错误恢复路径。

鲁棒性缺陷根源：关键异步提交缺少异常兜底和状态复位；错误提示依赖底层 API toast，不保证所有异常都有用户可见反馈。

优化修复方案：将 `handleSubmit` 包裹在 `try/catch/finally` 中，`finally` 必须恢复 `isSubmitting=false`；catch 中记录可诊断错误并展示统一提示；提交成功后再跳转，失败留在当前页保留表单内容；增加重试按钮或刷新资格状态。

### R-02 支付页支付失败后会永久停留在支付中

问题精准位置：`pages/payment/index.vue:125`、`pages/payment/index.vue:130`、`pages/payment/index.vue:133`

触发条件与复现步骤：进入支付页，点击“确认支付”；让 `markOrderPaid` 网络失败或云函数返回错误；观察支付按钮状态。

系统失效表现：`isPaying.value = true` 后没有 `try/catch/finally`，失败时按钮持续禁用，页面无恢复路径。

鲁棒性缺陷根源：异步状态锁未在异常路径释放；支付动作未设计失败态和重试态。

优化修复方案：为 `handlePay` 增加 `try/catch/finally`；失败时刷新订单状态，展示“支付未完成，请重试”；对重复点击使用幂等 key，而不是只靠前端禁用；支付成功以服务端状态为准再跳转。

### R-03 支付页加载失败缺少错误态，且默认活动 ID 会误创建订单

问题精准位置：`pages/payment/index.vue:99`、`pages/payment/index.vue:104`、`pages/payment/index.vue:110`

触发条件与复现步骤：访问 `/pages/payment/index` 不带 `activityId`；或携带不存在的活动 ID；或加载活动详情失败。

系统失效表现：页面默认使用 `'102'` 作为活动 ID 并调用 `ensureOrderForActivity`，可能创建错误订单；`loadData` 只有 `try/finally` 没有 catch，失败时页面 loading 消失但缺少明确错误提示和返回路径。

鲁棒性缺陷根源：路由参数缺失时使用演示兜底值；加载动作将“查询详情”和“创建订单”耦合，缺少失败分支。

优化修复方案：缺少 `activityId` 时立即显示错误并返回上一页；移除默认 `'102'`；先成功加载活动和报名资格，再由用户主动确认创建订单；`loadData` 增加 `errorState`、重试按钮和空数据保护。

### R-04 活动编辑保存失败后状态可能不一致且无异常恢复

问题精准位置：`pages/activity/edit.vue:407`、`pages/activity/edit.vue:421`

触发条件与复现步骤：进入活动编辑页，修改内容后断网保存；或云函数返回 FORBIDDEN/VALIDATION_ERROR。

系统失效表现：`handleSave` 没有 `try/catch/finally`，异常会冒泡；用户可能看到底层 toast 或无反馈；如果部分本地缓存已更新，页面和云端状态可能不一致。

鲁棒性缺陷根源：保存流程未区分校验失败、权限失败、网络失败、服务端失败；缺少保存中状态和回滚策略。

优化修复方案：保存前执行完整前端校验，保存时设置 `isSaving`；catch 中按错误码给出可恢复提示；保存成功后重新拉取云端详情覆盖本地状态；失败不更新本地缓存。

### R-05 管理台加载和批量通知缺少异常分支

问题精准位置：`pages/manage/dashboard.vue:277`、`pages/manage/dashboard.vue:315`、`pages/manage/dashboard.vue:349`

触发条件与复现步骤：活动管理页加载过程中 `getActivityDetail` 或 `listApplications` 任一接口失败；取消活动时 `createMessage` 某个通知失败；切换活动状态时云函数失败。

系统失效表现：页面可能部分数据为空但继续渲染；取消活动时状态已切换但部分用户未收到通知；异常未统一聚合展示，管理者无法知道哪些动作成功、哪些失败。

鲁棒性缺陷根源：多接口并行和批量操作缺少 `Promise.allSettled` 级别的部分失败处理；业务状态更新与通知发送缺少可靠队列。

优化修复方案：加载阶段使用独立错误态，活动详情失败则阻断管理页；列表失败允许重试；批量通知改为服务端异步任务或消息 outbox，状态切换成功后由后台保证最终发送；前端展示通知发送结果统计。

### R-06 运营台统计一次最多读取 1000 条，数据量增长后统计失真

问题精准位置：`uniCloud-aliyun/cloudfunctions/surego-moderation/index.js:217`

触发条件与复现步骤：活动、举报或用户数据超过 1000 条后访问运营台统计。

系统失效表现：统计结果只覆盖最近或默认查询的 1000 条数据，待审核数、举报数、隐藏数可能不准确；运营决策依赖错误数据。

鲁棒性缺陷根源：统计实现使用普通列表拉取和内存聚合，没有数据库聚合、分页累计或预聚合表。

优化修复方案：使用数据库聚合 `groupBy/count` 或按状态分别 `count()`；为运营统计建立定时预聚合表；接口返回 `isPartial` 或 `updatedAt`；避免 `limit(1000)` 作为统计边界。

### R-07 报名提交并发会产生重复报名

问题精准位置：`uniCloud-aliyun/cloudfunctions/surego-application/index.js:70`、`uniCloud-aliyun/cloudfunctions/surego-application/index.js:134`、`uniCloud-aliyun/cloudfunctions/surego-application/index.js:154`

触发条件与复现步骤：同一用户在弱网下连续点击报名，或两个客户端同时提交同一活动报名。

系统失效表现：两个请求同时执行 `getExistingApplication` 都查不到记录，随后都 `add` 成功，形成重复报名；后续成员列表、审核、签到、订单关联均可能异常。

鲁棒性缺陷根源：先查后写不是原子操作，数据库无唯一索引或幂等键。

优化修复方案：对 `surego-applications` 增加唯一键 `activity_id + user_id`；提交接口使用幂等 key；数据库插入捕获唯一约束冲突后返回已有记录；前端禁用按钮只是体验优化，不作为并发控制。

### R-08 报名审核并发导致参与人数计数错乱

问题精准位置：`uniCloud-aliyun/cloudfunctions/surego-application/index.js:210`、`uniCloud-aliyun/cloudfunctions/surego-application/index.js:220`、`uniCloud-aliyun/cloudfunctions/surego-application/index.js:228`、`uniCloud-aliyun/cloudfunctions/surego-application/index.js:234`

触发条件与复现步骤：两个管理员页面或重复点击同时审核同一报名；或一个请求通过、另一个请求拒绝/撤销。

系统失效表现：`participantCount` 可能重复加一、重复减一或减成负数；报名状态和活动人数不一致。

鲁棒性缺陷根源：读取旧状态、更新报名、递增活动计数不是事务；没有状态版本号和条件更新。

优化修复方案：使用数据库事务：读取报名当前状态，条件更新 `_id + status == oldStatus`，再根据状态差更新活动人数；活动人数不得小于 0 且不得超过上限；接口返回冲突错误 `STATUS_CONFLICT`，前端刷新数据。

### R-09 订单 `ensureForActivity` 并发会产生重复订单

问题精准位置：`uniCloud-aliyun/cloudfunctions/surego-order/index.js:117`、`uniCloud-aliyun/cloudfunctions/surego-order/index.js:143`、`uniCloud-aliyun/cloudfunctions/surego-order/index.js:160`

触发条件与复现步骤：用户快速多次进入支付页，或多个终端同时打开同一活动支付页。

系统失效表现：多个 pending 订单被创建；后续支付、退款、关闭订单时只处理其中一个，订单列表出现重复记录。

鲁棒性缺陷根源：`findOrderByActivity` 与 `add` 非原子；无 `activity_id + user_id` 唯一索引；支付页进入即创建订单放大并发概率。

优化修复方案：订单表增加唯一索引；`ensureForActivity` 改为原子 upsert 或捕获唯一键冲突后查询已有订单；引入订单 `idempotency_key`；前端只在用户确认购买时调用创建订单。

### R-10 签到确认并发会产生重复签到或重复消息

问题精准位置：`uniCloud-aliyun/cloudfunctions/surego-checkin/index.js:190`、`uniCloud-aliyun/cloudfunctions/surego-checkin/index.js:209`、`uniCloud-aliyun/cloudfunctions/surego-checkin/index.js:217`

触发条件与复现步骤：局长扫码后重复点击确认，或多个管理端同时扫描同一成员二维码。

系统失效表现：两个请求同时未查到已有签到，随后都写入记录；消息中心可能重复收到签到确认通知；签到统计数偏大。

鲁棒性缺陷根源：查重和写入不是事务；缺少 `activity_id + user_id` 唯一索引；消息创建也缺少强幂等约束。

优化修复方案：签到表增加唯一索引；确认接口以事务写入签到并捕获冲突返回已有记录；消息表对 `user_id + event_key` 增唯一索引；前端确认按钮在请求期间禁用并刷新列表。

### R-11 消息 eventKey 幂等依赖先查后写，仍可能重复

问题精准位置：`uniCloud-aliyun/cloudfunctions/surego-message/index.js:100`、`uniCloud-aliyun/cloudfunctions/surego-message/index.js:103`、`common/api/message.js:35`

触发条件与复现步骤：同一业务事件同时触发两次消息创建，例如审核通过和活动取消批量通知重试。

系统失效表现：两个请求同时查询不到相同 `event_key`，随后均插入，消息重复。

鲁棒性缺陷根源：应用层幂等不具备并发安全；数据库无唯一约束。

优化修复方案：消息表建立唯一索引 `user_id + event_key`；插入冲突时查询并返回已有消息；对没有 `event_key` 的系统消息强制生成稳定业务键。

### R-12 camelCase 与 snake_case 混用导致查询遗漏

问题精准位置：`uniCloud-aliyun/cloudfunctions/surego-application/index.js:197`、`uniCloud-aliyun/cloudfunctions/surego-order/index.js:263`、`uniCloud-aliyun/cloudfunctions/surego-order/index.js:270`、`uniCloud-aliyun/cloudfunctions/surego-activity/index.js:186`

触发条件与复现步骤：数据库中同时存在历史 camelCase 记录和新 snake_case 记录；调用列表接口或详情接口。

系统失效表现：`listByActivity` 查询 `{activityId}` 可能漏掉只含 `activity_id` 的记录；订单列表按 `{userId}` 查询漏掉只含 `user_id` 的记录；应用列表需要手动查两套字段，维护成本高。

鲁棒性缺陷根源：数据模型迁移未完成，读写层没有统一字段规范；兼容逻辑散落在各云函数。

优化修复方案：制定统一数据库字段规范，建议服务端存储全部 snake_case；编写一次性迁移脚本补齐字段并清理旧字段；所有查询只使用规范字段；API 输出层负责转换 camelCase 给前端。

### R-13 活动更新未使用版本号，弱网重试会覆盖新数据

问题精准位置：`uniCloud-aliyun/cloudfunctions/surego-activity/index.js:245`、`uniCloud-aliyun/cloudfunctions/surego-activity/index.js:251`、`pages/activity/edit.vue:407`

触发条件与复现步骤：创建者在两个设备同时编辑同一活动；A 保存后，B 基于旧数据保存。

系统失效表现：B 的旧表单覆盖 A 的最新修改，用户无冲突提示；参与者看到活动详情突然回退。

鲁棒性缺陷根源：更新接口没有 `updated_at` 条件、版本号或字段级 patch 合并策略。

优化修复方案：活动表增加 `version` 字段；编辑页加载时保存版本；更新时执行条件更新 `_id + version`，成功后 `version + 1`；冲突时返回 `VERSION_CONFLICT`，前端提示用户重新加载并展示差异。

### R-14 活动详情和报名页加载失败缺少完整错误态

问题精准位置：`pages/activity/register.vue:208`、`pages/activity/register.vue:221`、`pages/activity/detail.vue:419`

触发条件与复现步骤：打开活动详情或报名页时活动不存在、接口超时、网络断开或活动被删除。

系统失效表现：页面可能展示默认空字段、按钮不可用但原因不清晰；报名页 `finally` 停止 loading 后没有统一 errorState；用户无法重试或返回。

鲁棒性缺陷根源：页面状态模型只有 `loading` 和正常数据，缺少 `error/notFound/forbidden/offline` 等状态。

优化修复方案：为详情和报名页建立状态机：`loading -> ready | not_found | forbidden | offline | error`；每个错误态提供重试和返回入口；所有计算属性先判断 `activity.value` 是否存在。

### R-15 活动创建和编辑对数值边界处理不足

问题精准位置：`pages/activity/create.vue:388`、`pages/activity/create.vue:417`、`pages/activity/edit.vue:248`、`common/api/activity.js:461`

触发条件与复现步骤：输入 `maxParticipants=0`、负数、超大数、非数字字符串；输入金额为小数、极大数或科学计数法；编辑页直接保存。

系统失效表现：人数容量、金额、问题列表等字段可能变为 `0`、默认值或异常大值；页面展示、订单金额、审核和报名容量判断异常。

鲁棒性缺陷根源：前端校验只覆盖部分必填项；`Number(value) || default` 会吞掉非法值并静默替换；服务端缺少最终校验。

优化修复方案：建立共享校验函数 `validateActivityPayload`；人数必须是 1-500 整数，金额使用分为单位的非负整数，上限明确；非法输入返回字段级错误；编辑页复用创建页完整校验。

### R-16 `Number(payload.amount) || 0` 会把非法金额静默改为 0

问题精准位置：`uniCloud-aliyun/cloudfunctions/surego-order/index.js:103`、`common/api/order.js:73`、`common/api/activity.js:490`

触发条件与复现步骤：构造订单或活动 payload，传入 `amount:''`、`amount:'abc'`、`amount:null`、`amount:NaN`。

系统失效表现：非法金额被静默转成 0，后续流程无法区分“免费活动”和“异常金额”。

鲁棒性缺陷根源：使用 `Number(x) || 0` 混淆非法值和合法 0；没有金额类型和精度规范。

优化修复方案：金额统一使用整数分 `amount_cent`；使用 `Number.isInteger(amountCent) && amountCent >= 0` 校验；非法值直接返回 `VALIDATION_ERROR`，不做默认替换；前端显示层再格式化为元。

### R-17 云函数参数缺少统一 schema 校验，异常输入会进入数据库操作

问题精准位置：`uniCloud-aliyun/cloudfunctions/surego-activity/index.js:153`、`uniCloud-aliyun/cloudfunctions/surego-application/index.js:124`、`uniCloud-aliyun/cloudfunctions/surego-order/index.js:124`、`uniCloud-aliyun/cloudfunctions/surego-checkin/index.js:177`、`uniCloud-aliyun/cloudfunctions/surego-moderation/index.js:128`

触发条件与复现步骤：直接调用云函数，传入缺失 `id/activityId`、数组替代字符串、超长字符串、异常状态枚举或空 payload。

系统失效表现：可能返回不稳定错误、数据库查询空结果、写入异常字段、前端展示 `undefined`，甚至触发云函数未捕获异常。

鲁棒性缺陷根源：每个 action 自行读取 payload，缺少统一 DTO/schema、字段投影和错误码。

优化修复方案：引入轻量校验器，例如 `validate(schema, payload)`；每个 action 定义必填字段、类型、长度、枚举和默认值；校验失败统一返回 `VALIDATION_ERROR` 和字段错误列表；数据库写入只使用校验后的 DTO。

### R-18 云函数缺少全局异常捕获和 traceId

问题精准位置：`uniCloud-aliyun/cloudfunctions/surego-activity/index.js:153`、`uniCloud-aliyun/cloudfunctions/surego-user/index.js:341`、`uniCloud-aliyun/cloudfunctions/surego-moderation/index.js:128`、`common/api/cloud.js:49`

触发条件与复现步骤：数据库连接异常、集合不存在、第三方服务异常或代码抛出未捕获错误。

系统失效表现：前端接收到平台级错误或原始 message，无法区分可重试、需登录、需刷新、服务端故障；服务端日志和用户报错无法关联。

鲁棒性缺陷根源：云函数主入口没有 `try/catch` 包装；缺少标准响应协议和 traceId。

优化修复方案：所有云函数入口使用 `withHandler(async (ctx)=>{})` 包装，自动生成 `traceId`、捕获异常、映射错误码、写结构化日志；前端根据错误码处理登录失效、限流、校验失败和系统错误。

### R-19 mock fallback 与云端模式共存，弱网时数据一致性不可控

问题精准位置：`common/config/runtime.js:1`、`common/config/runtime.js:3`、`common/api/cloud.js:70`、`common/api/auth.js:237`

触发条件与复现步骤：切换 `APP_MODE` 到非 trial 或误配置 `ALLOW_MOCK_FALLBACK`；云函数超时或不可用；前端 API 进入本地 fallback。

系统失效表现：用户以为操作成功，但数据只写入本地 storage；云端恢复后列表、报名、订单、消息与本地状态不一致。

鲁棒性缺陷根源：本地 mock 与真实云端共享同一业务 API，没有显式离线队列、同步状态或冲突处理。

优化修复方案：生产和试运行环境彻底关闭业务写操作 fallback；开发模式 UI 明确标识 mock；若要支持离线，设计本地 outbox、同步状态、冲突解决和用户可见的“待同步”标识。

### R-20 本地 storage 缓存无容量、版本和迁移策略

问题精准位置：`common/api/activity.js:441`、`common/api/application.js:9`、`common/api/order.js:10`、`common/api/message.js:8`、`common/api/moderation.js:22`

触发条件与复现步骤：长期使用 App、创建大量活动/报名/消息，或版本升级导致缓存字段结构变化。

系统失效表现：storage 体积膨胀，读取变慢或失败；旧字段导致页面显示异常；本地缓存覆盖新接口返回值。

鲁棒性缺陷根源：缓存没有版本号、TTL、最大条数、迁移函数和清理机制。

优化修复方案：所有缓存 key 增加版本，如 `surego_v2_*`；列表缓存限制最大条数和过期时间；App 启动时执行缓存迁移；云端数据优先，缓存只用于展示占位或离线只读。

### R-21 列表接口缺少分页游标，数据量增长后页面性能和完整性下降

问题精准位置：`uniCloud-aliyun/cloudfunctions/surego-activity/index.js:160`、`uniCloud-aliyun/cloudfunctions/surego-activity/index.js:184`、`uniCloud-aliyun/cloudfunctions/surego-application/index.js:197`、`uniCloud-aliyun/cloudfunctions/surego-order/index.js:270`

触发条件与复现步骤：单用户创建或参与活动超过 100 条，活动报名人数超过 100，订单超过默认 limit。

系统失效表现：列表截断但前端不知晓；滚动加载无法继续获取后续数据；管理页成员或订单不完整。

鲁棒性缺陷根源：接口只使用固定 limit，没有返回 `nextCursor/hasMore`；前端页面没有分页状态。

优化修复方案：列表接口统一支持 `limit + cursor` 或 `pageToken`；按 `created_at/_id` 组合游标排序；响应返回 `items/nextCursor/hasMore`；前端实现分页加载和刷新合并。

### R-22 角色更新的最后管理员保护逻辑不可靠

问题精准位置：`uniCloud-aliyun/cloudfunctions/surego-user/index.js:298`、`uniCloud-aliyun/cloudfunctions/surego-user/index.js:417`、`uniCloud-aliyun/cloudfunctions/surego-user/index.js:428`

触发条件与复现步骤：系统中管理员角色字段以数组形式存储；运营同时修改两个管理员角色；或 role 字段结构不符合 `where({ role: db.command.in(['admin']) })` 预期。

系统失效表现：可能错误判断管理员数量，导致最后一个管理员被移除，或阻止合法修改。

鲁棒性缺陷根源：管理员计数查询对数组字段匹配依赖不明确；更新角色没有事务和条件版本。

优化修复方案：统一 `uni-id-users.role` 为数组并验证查询语义；增加系统配置表记录超级管理员；角色更新使用事务并在事务内重新计数；禁止操作当前最后管理员。

### R-23 `createCode` 返回的签到码与 `confirm` 期望码不一致

问题精准位置：`uniCloud-aliyun/cloudfunctions/surego-checkin/index.js:179`、`uniCloud-aliyun/cloudfunctions/surego-checkin/index.js:209`、`common/api/checkin.js:58`

触发条件与复现步骤：调用 `createCheckinCode(activityId)` 获取随机码，再调用 `confirmCheckin` 使用该码核销。

系统失效表现：随机码格式有效但无法通过 `confirm` 的确定性码比较，导致功能看似可用、实际无法核销。

鲁棒性缺陷根源：生成码和校验码使用两套完全不同算法；`expiresIn` 只返回给客户端但没有服务端状态。

优化修复方案：统一生成和校验模型；如果使用一次性随机码，必须落库并由 `confirm` 查库；如果使用参与者固定码，删除 `createCode` 或改名，避免误用。

### R-24 页面权限提示不阻断后续页面渲染

问题精准位置：`pages/manage/dashboard.vue:286`、`pages/manage/checkin.vue:191`、`pages/activity/edit.vue:356`

触发条件与复现步骤：非创建者通过分享路径或手动路由进入活动管理、签到、编辑页。

系统失效表现：页面展示部分管理信息、按钮可能不可用或后端拒绝；用户体验混乱，且错误路径没有返回。

鲁棒性缺陷根源：权限检查只 toast，不设置 `forbidden` 状态、不 redirect、不阻断页面后续逻辑。

优化修复方案：加载活动后若无权限，设置 `pageState='forbidden'` 并隐藏全部管理数据；提供返回活动详情按钮；所有操作按钮根据状态机渲染。

### R-25 依赖服务不可用时缺少降级和超时策略

问题精准位置：`common/api/cloud.js:19`、`common/api/location.js:56`、`common/api/upload.js:64`

触发条件与复现步骤：uniCloud 网络超时、定位服务拒绝授权、上传服务异常。

系统失效表现：页面只显示通用失败提示，缺少重试、取消、手动输入、排队上传或服务状态提示；用户容易重复操作。

鲁棒性缺陷根源：基础 API 封装没有统一超时、重试、取消和错误分类；页面各自处理失败。

优化修复方案：为云函数调用加超时包装和错误分类；可重试错误使用指数退避；定位失败进入手动城市/地址输入；上传失败保留本地选择并支持重新上传；所有长耗时动作支持取消。

## 三、优先级修复路线

第一优先级：修复关键页面 `try/catch/finally`、报名/订单/签到唯一约束与事务、金额和活动 payload 校验。

第二优先级：统一字段命名和数据迁移、建立云函数全局异常包装、修复统计分页与列表游标。

第三优先级：清理 mock fallback、完善缓存版本、增强依赖服务超时和降级体验。
