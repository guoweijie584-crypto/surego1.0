# 项目安全性问题深度审计报告

审计日期：2026-05-06  
审计范围：`D:\SureGo3\surego1.0` 当前项目全量源码、`pages/` 前端页面、`common/api/` 前端 API 封装、`components/` 组件、`uniCloud-aliyun/cloudfunctions/` 云函数、`uniCloud-aliyun/database/` 数据库 schema、`manifest.json`、依赖锁文件。  
报告边界：本报告仅记录安全性问题，不记录鲁棒性缺陷和业务闭环缺陷。

## 一、总体结论

当前项目的核心安全短板集中在 5 类：云函数鉴权模型未统一使用 `uniID.checkToken`，多个高危接口仅校验“传入 userId 对应 token 数组中是否包含 token”；`user-center` 保留了可被任意登录用户调用的账户与权限管理能力；数据库 schema 中存在公开读写的演示表和过宽的 SureGo 业务表权限；订单、报名、签到等关键动作过度信任客户端传参；客户端本地存储、WebView、上传、App 权限和日志存在数据泄露或滥用风险。

风险等级统计：高危 10 项，中危 8 项，低危 2 项。高危问题应优先修复后再进行生产发布或线上试运行。

## 二、安全问题清单

### S-01 `user-center` 云函数存在账户和权限管理越权接口

问题精准位置：`uniCloud-aliyun/cloudfunctions/user-center/index.js:12`、`uniCloud-aliyun/cloudfunctions/user-center/index.js:22`、`uniCloud-aliyun/cloudfunctions/user-center/index.js:90`、`uniCloud-aliyun/cloudfunctions/user-center/index.js:96`、`uniCloud-aliyun/cloudfunctions/user-center/index.js:125`、`uniCloud-aliyun/cloudfunctions/user-center/index.js:140`

风险等级：高

风险原理与触发条件：`noCheckAction` 将 `encryptPwd`、`sendSmsCode` 等敏感能力放入免鉴权列表；非免鉴权动作只执行 `uniID.checkToken(event.uniIdToken)`，没有校验调用者是否为管理员、是否拥有具体权限、是否只能操作自己。任意已登录用户可调用 `resetPwd` 将任意 `params.uid` 密码重置为固定值 `123456`，也可调用 `addRole`、`addPermission`、`bindRole`、`bindPermission` 修改权限体系。

潜在安全危害：账号接管、管理员权限伪造、权限系统被污染、短信接口被滥用、固定默认密码造成批量撞库风险。

可落地修复方案：删除或下线该旧版 `user-center` 云函数，保留 `uni-id-co` 官方对象式云函数作为认证入口；如必须保留，新增统一鉴权中间件，所有管理类 action 必须满足 `payload.role.includes('admin')` 或显式权限，如 `user:reset-password`、`role:write`；`resetPwd` 必须仅允许本人通过旧密码/短信验证码/邮箱验证码重置，管理员重置必须生成一次性随机临时密码并强制首次登录修改；`encryptPwd` 不应对外暴露；`sendSmsCode` 加入图形验证码、频率限制、手机号黑名单和 IP 限流。

### S-02 SureGo 业务云函数未统一使用 `uniID.checkToken` 校验 token 有效期、账号状态和角色实时性

问题精准位置：`uniCloud-aliyun/cloudfunctions/surego-activity/index.js:44`、`uniCloud-aliyun/cloudfunctions/surego-application/index.js:25`、`uniCloud-aliyun/cloudfunctions/surego-order/index.js:24`、`uniCloud-aliyun/cloudfunctions/surego-checkin/index.js:29`、`uniCloud-aliyun/cloudfunctions/surego-message/index.js:26`、`uniCloud-aliyun/cloudfunctions/surego-moderation/index.js:34`、`uniCloud-aliyun/cloudfunctions/surego-user/index.js:240`

风险等级：高

风险原理与触发条件：多个云函数通过读取 `uni-id-users.token` 数组并判断 `event.uniIdToken` 是否存在来确认身份，没有调用 `uniID.checkToken` 或 `uni-id-common.checkToken`。该方式无法稳定校验 token 签名、过期时间、账号禁用状态、`valid_token_date`、权限变更后的 token 失效策略，也无法统一处理自动刷新 token。

潜在安全危害：过期 token 或已被降权用户仍可能访问业务接口；账号被封禁、角色被回收后，旧 token 只要仍存在数据库数组就可能继续操作；所有业务云函数各自实现鉴权，容易出现绕过和行为不一致。

可落地修复方案：新增 `uniCloud-aliyun/cloudfunctions/common/surego-auth/index.js`，内部使用 `uni-id-common` 或 `uniID.checkToken(event.uniIdToken)` 返回 `uid/role/permission`；所有 SureGo 云函数删除本地 `isTokenOwnedByUser/resolveUserContext` 重复实现，统一调用 `requireAuth(event)`、`requireRole(event, ['admin','operator'])`、`requireOwner(resourceOwnerId)`；每次接口操作以 checkToken 返回的 `uid` 为准，禁止从 `event.userId`、`payload.userId` 反推调用人。

### S-03 前端 API 将 `userId`、`roles` 和本地用户资料随请求发送，形成鉴权绕过诱导面

问题精准位置：`common/api/cloud.js:19`、`common/api/cloud.js:30`、`common/api/cloud.js:33`、`common/api/auth.js:145`、`common/api/auth.js:155`、`common/api/auth.js:176`

风险等级：中

风险原理与触发条件：`callSuregoFunction` 将 `userId: payload.userId || payload.user_id || userId` 和 `roles: profile.role || profile.roles || []` 写入云函数 event。`getCurrentUserId`、`isLoggedIn`、`isOpsUser` 又从本地 storage 或模拟登录状态读取身份信息。攻击者可以篡改客户端本地 storage、Hook `uniCloud.callFunction` 请求体，诱导后端错误信任客户端身份字段。

潜在安全危害：一旦后端某个接口遗漏 token 绑定校验，就会被本地身份字段直接绕过；前端 ops 菜单、页面可见性和按钮状态不能作为权限边界；日志和异常中也会留下可伪造身份轨迹。

可落地修复方案：`common/api/cloud.js` 只传 `action`、`payload`、`uniIdToken`，不传 `userId`、`roles`、`profile`；后端统一从 `checkToken` 获取真实 `uid/role/permission`；前端 `isOpsUser` 只作为 UI 展示优化，所有业务接口必须后端二次鉴权；试运行模式下禁用 mock 身份参与任何云端请求。

### S-04 用户资料接口存在水平越权和隐私数据泄露

问题精准位置：`uniCloud-aliyun/cloudfunctions/surego-user/index.js:348`、`uniCloud-aliyun/cloudfunctions/surego-user/index.js:397`、`uniCloud-aliyun/database/surego-users.schema.json:5`

风险等级：高

风险原理与触发条件：`profile` action 允许传入 `payload.userId` 查询任意用户 SureGo profile；`getProfiles` 允许任意登录用户传入任意 `userIds` 批量读取完整 profile；schema 又配置 `surego-users` 只要登录即可 read。返回数据包含 `roles`、`credit_score`、`profile_completed_at`、昵称、头像、简介等字段。

潜在安全危害：用户隐私数据被批量枚举，运营角色和信用信息泄露，攻击者可建立用户画像并辅助后续钓鱼、越权或社工攻击。

可落地修复方案：`profile` 只允许读取本人完整资料，读取他人必须走 `publicProfile` 且只返回 `userId/nickname/avatar/bio` 等公开字段；`getProfiles` 仅用于业务必要场景，按活动成员关系校验调用者是否为活动创建者或同活动成员；schema 将 `surego-users.read` 改为 `doc.user_id == auth.uid || 'admin' in auth.role || 'operator' in auth.role`，并对公开字段拆分 `surego-public-profiles` 或通过云函数投影返回。

### S-05 数据库 schema 权限过宽，ClientDB 可绕过云函数业务过滤

问题精准位置：`uniCloud-aliyun/database/surego-activities.schema.json:5`、`uniCloud-aliyun/database/surego-reports.schema.json:5`、`uniCloud-aliyun/database/surego-checkins.schema.json:6`、`uniCloud-aliyun/database/surego-orders.schema.json:6`、`uniCloud-aliyun/database/test.schema.json:6`、`uniCloud-aliyun/database/order.schema.json:6`、`uniCloud-aliyun/database/user-info.schema.json:6`、`uniCloud-aliyun/database/validate-demo.schema.json:6`

风险等级：高

风险原理与触发条件：`surego-activities.read` 为 `true`，可直接读取审核中、隐藏、驳回活动和创建者内部字段；`surego-reports.read` 为任意登录用户可读；`surego-checkins.create`、`surego-orders.create` 只要求登录，不约束 `user_id` 与 `auth.uid`；多个演示表 `test/order/user-info/validate-demo/permission-test-*` 存在公开读写。若 App 或 H5 保留 ClientDB 能力，攻击者可绕开云函数筛选直接访问数据库。

潜在安全危害：未审核活动泄露、举报内容泄露、伪造订单/签到记录、演示表被写入恶意数据或成为持久化载荷存储点。

可落地修复方案：生产环境删除演示 schema 和演示页面；所有 SureGo 业务表默认关闭 ClientDB 写入，改为云函数服务端写；保留 ClientDB 的表必须加字段级权限，例如 `create: auth.uid == doc.user_id`，`read: doc.user_id == auth.uid || doc.creator_id == auth.uid || role in ops`；活动表拆分公开列表视图与后台管理视图，公开读只返回 `moderation_status == 'approved' && status in ['recruiting','formed','ended']` 的非敏感字段。

### S-06 报名接口信任客户端状态并缺少活动准入校验

问题精准位置：`uniCloud-aliyun/cloudfunctions/surego-application/index.js:124`、`uniCloud-aliyun/cloudfunctions/surego-application/index.js:142`、`uniCloud-aliyun/cloudfunctions/surego-application/index.js:152`、`common/api/application.js:47`、`common/api/application.js:163`

风险等级：高

风险原理与触发条件：`submit` 使用客户端传入 payload 构造报名记录，并设置 `status: payload.status || 'pending'`。前端 `buildApplication` 会根据传入参数生成 `approved` 或 `pending`，攻击者也可直接传 `status:'approved'`。云端未基于活动真实 `require_approval`、`status`、`moderation_status`、报名截止时间、容量上限进行重新计算。

潜在安全危害：用户可绕过审核直接成为已通过成员；可报名未审核、隐藏、已结束或已满活动；恶意报名污染成员列表、订单和签到流程。

可落地修复方案：`submit` 中先读取活动记录并执行服务端准入校验：活动必须审核通过、处于可报名状态、未截止、未满员、非创建者；报名状态由服务端根据活动 `require_approval` 决定，客户端传入的 `status/requireApproval` 全部忽略；对同一 `activity_id + user_id` 建唯一索引；容量检查和写入报名使用事务或云数据库事务函数完成。

### S-07 订单支付、退款、关闭状态完全由客户端驱动

问题精准位置：`uniCloud-aliyun/cloudfunctions/surego-order/index.js:94`、`uniCloud-aliyun/cloudfunctions/surego-order/index.js:103`、`uniCloud-aliyun/cloudfunctions/surego-order/index.js:191`、`uniCloud-aliyun/cloudfunctions/surego-order/index.js:219`、`uniCloud-aliyun/cloudfunctions/surego-order/index.js:240`、`pages/payment/index.vue:133`

风险等级：高

风险原理与触发条件：订单金额 `amount`、类型 `type`、标题等由客户端 payload 提供；`markPaid` 只校验订单归属后直接将状态改为 `paid`；`refund` 和 `close` 也允许订单所有者直接修改状态，没有支付渠道回调、签名验签、交易号、幂等号、管理员审核或资金侧确认。

潜在安全危害：用户可零元购或篡改金额后获得付费活动资格；可自行伪造支付成功、退款成功、关闭订单；财务记录与真实收款不一致。

可落地修复方案：订单创建金额必须由服务端读取活动价格生成，客户端只能传 `activityId`；支付成功只能由支付平台异步回调云函数验签后更新，`markPaid` 禁止对普通客户端开放；退款必须由运营/支付回调状态机驱动，普通用户只能提交退款申请；订单表增加 `pay_trade_no`、`refund_trade_no`、`amount_cent`、`currency`、`status_version`、`idempotency_key`。

### S-08 签到凭证可预测且服务端不校验一次性与过期时间

问题精准位置：`common/api/checkin.js:42`、`uniCloud-aliyun/cloudfunctions/surego-checkin/index.js:168`、`uniCloud-aliyun/cloudfunctions/surego-checkin/index.js:179`、`uniCloud-aliyun/cloudfunctions/surego-checkin/index.js:185`、`uniCloud-aliyun/cloudfunctions/surego-checkin/index.js:209`

风险等级：高

风险原理与触发条件：参与者核销码由 `activityId:userId` 做确定性哈希生成，缺少服务端随机数和签名密钥；`createCode` 返回随机 `SGxxxxxx` 和 `expiresIn:300`，但该随机码没有保存，也不被 `confirm` 使用；`confirm` 只比较确定性 `expectedCode`，没有一次性 token、过期时间、场次时间窗口和重放控制。

潜在安全危害：只要获得或枚举活动 ID 与用户 ID，就可构造或复用签到码；截图泄露后长期可用；历史二维码可被重复出示直到数据库已有记录才被阻止。

可落地修复方案：创建 `surego-checkin-codes` 表，服务端生成 `nonce + expires_at + activity_id + user_id + HMAC`，二维码只包含短 token；`confirm` 读取 token 记录并校验未过期、未使用、活动时间窗口、参与资格和支付状态，然后在事务中标记 used 并写入签到；删除或重写当前未落库的 `createCode`。

### S-09 消息创建接口允许任意登录用户向任意用户写通知

问题精准位置：`uniCloud-aliyun/cloudfunctions/surego-message/index.js:76`、`uniCloud-aliyun/cloudfunctions/surego-message/index.js:98`、`uniCloud-aliyun/cloudfunctions/surego-message/index.js:100`

风险等级：中

风险原理与触发条件：`surego-message` 的 `create` action 对所有登录用户开放，`buildRecord` 直接使用 `payload.userId` 作为消息接收人。接口没有校验消息类型是否允许当前调用者创建，也没有校验业务上下文，例如是否为活动创建者、是否为运营系统、是否为订单或审核流程触发。

潜在安全危害：恶意用户可向任意用户发送系统通知、伪造审核/支付/运营消息、进行钓鱼或骚扰；消息中心可信度下降。

可落地修复方案：禁止普通客户端直接调用通用 `create`；消息创建应只作为其他云函数内部服务能力，按 `eventType` 做白名单校验；若需要用户私信，单独设计会话与反骚扰规则；所有系统消息由服务端根据真实业务状态生成，`recipient_id` 不接受客户端任意指定。

### S-10 活动创建和编辑存在服务端批量赋值与内容校验缺失

问题精准位置：`uniCloud-aliyun/cloudfunctions/surego-activity/index.js:138`、`uniCloud-aliyun/cloudfunctions/surego-activity/index.js:224`、`uniCloud-aliyun/cloudfunctions/surego-activity/index.js:245`、`pages/activity/create.vue:388`、`pages/activity/edit.vue:407`

风险等级：中

风险原理与触发条件：云端 `withoutEmptyId` 仅删除少量 moderation 字段，然后将 payload 大量写入数据库；缺少服务端字段白名单、类型校验、长度限制、枚举校验、问题数量限制、图片 URL 校验和敏感词/违规内容校验。前端校验只能约束正常用户，不能抵御直接调用云函数。

潜在安全危害：攻击者可写入异常字段污染数据模型；超长内容造成页面性能问题；恶意链接、异常图片地址、违规文本进入公开活动详情；编辑已通过活动后可能将安全内容替换为违规内容。

可落地修复方案：云函数建立活动字段白名单和校验器，例如 `title` 1-40 字、`description` 1-300 字、`questions` 最多 5 个且每个 50 字、`amount_cent` 非负整数、`max_participants` 1-500；更新接口只允许业务字段；所有创建/编辑统一进入内容审核，已通过活动编辑关键内容后自动重置为 `reviewing/pending`。

### S-11 WebView 页面允许加载任意 URL

问题精准位置：`pages/webview/webview.vue:3`、`pages/webview/webview.vue:9`

风险等级：中

风险原理与触发条件：页面从路由参数读取 `url` 并直接绑定到 `<web-view :src="url">`，没有协议限制、域名白名单、URL 解码校验和标题可信度校验。用户点击构造链接即可在 App 内打开任意外部页面。

潜在安全危害：App 内钓鱼、仿冒登录、诱导支付、加载恶意 H5；用户误认为页面属于 SureGo 官方；若平台 WebView 与 JSBridge 配置不当，可能扩大为本地能力滥用。

可落地修复方案：仅允许 `https://` 且域名在 `ALLOWED_WEBVIEW_HOSTS` 中；对路由参数执行 `decodeURIComponent` 后使用 `new URL` 严格解析；禁止 `javascript:`、`file:`、`data:`、内网地址和 IP 字面量；页面顶部展示真实域名；外部未知链接调用系统浏览器而非内嵌 WebView。

### S-12 上传能力缺少文件类型、大小和内容校验

问题精准位置：`common/api/upload.js:25`、`common/api/upload.js:46`、`common/api/upload.js:63`

风险等级：中

风险原理与触发条件：`chooseImageFile` 仅调用 `uni.chooseImage`，`uploadImageFile` 未校验文件大小、扩展名白名单、MIME、图片魔数和实际内容；`buildCloudPath` 直接从本地路径推断后缀；`uniCloud.uploadFile` 使用 `cloudPathAsRealPath: true`。

潜在安全危害：超大图片耗尽存储和流量；伪装扩展名文件进入云存储；恶意图片或 SVG/HTML 被后续页面展示时形成内容注入风险；路径策略可预测导致资源枚举概率上升。

可落地修复方案：上传前限制大小，例如头像小于 2MB、活动图小于 5MB；只允许 `jpg/jpeg/png/webp`；服务端或上传前读取文件头校验魔数；云存储路径使用 `env/userId/yyyy/mm/randomUUID.ext`；上传后生成只读 fileID，前端展示使用平台安全图片组件，禁止直接渲染 SVG/HTML。

### S-13 认证 token 和用户资料明文保存在本地 storage

问题精准位置：`common/api/auth.js:85`、`common/api/auth.js:266`、`common/api/auth.js:269`、`common/api/auth.js:271`、`common/api/auth.js:273`、`common/api/cloud.js:13`

风险等级：中

风险原理与触发条件：`uni_id_token`、过期时间、`uni_id_user`、`surego_current_user` 等保存在本地 storage，任何运行在同一客户端上下文的恶意代码、调试工具、被篡改包或越狱/Root 环境都可能读取或篡改。

潜在安全危害：token 被窃取后可冒用用户身份；本地角色被篡改会影响前端可见功能；用户资料泄露；长期 token 增加被盗后的有效攻击窗口。

可落地修复方案：缩短 token 有效期并启用刷新 token 轮换；敏感平台使用安全存储能力或 Keychain/Keystore 插件；登出、改密、封禁、角色变更后服务端设置 `valid_token_date` 使旧 token 失效；前端不持久化角色权限，只缓存非敏感展示字段；所有权限以后端为准。

### S-14 `manifest.json` 配置过多高敏 Android 权限且微信小程序关闭 URL 校验

问题精准位置：`manifest.json:30`、`manifest.json:36`、`manifest.json:49`、`manifest.json:50`、`manifest.json:52`、`manifest.json:55`、`manifest.json:56`、`manifest.json:59`、`manifest.json:61`、`manifest.json:62`、`manifest.json:85`

风险等级：高

风险原理与触发条件：App 权限包含联系人、短信、通话、日志、系统悬浮窗、系统设置、模拟定位、文件系统挂载等明显超出活动报名产品需要的能力；微信小程序 `urlCheck:false` 降低域名访问约束。

潜在安全危害：应用商店审核失败；用户隐私合规风险显著上升；一旦包体或三方 SDK 被滥用，可读取短信、联系人、日志或诱导系统级权限；小程序网络访问域名治理失效。

可落地修复方案：最小化权限，只保留定位、相册/相机、网络等确有业务必要的权限；删除 `READ_SMS/SEND_SMS/READ_CONTACTS/WRITE_CONTACTS/READ_LOGS/SYSTEM_ALERT_WINDOW/WRITE_SETTINGS/ACCESS_MOCK_LOCATION` 等高危权限；补充隐私政策和权限使用弹窗；小程序生产环境开启 `urlCheck:true` 并配置合法域名。

### S-15 日志输出泄露密钥、事件参数和调试信息

问题精准位置：`uniCloud-aliyun/cloudfunctions/cloud-object-demo/index.obj.js:63`、`uniCloud-aliyun/cloudfunctions/use-common/index.js:9`、`uniCloud-aliyun/cloudfunctions/secure-network/index.js:5`、`uniCloud-aliyun/cloudfunctions/secure-network-object/index.obj.js:11`、`common/api/cloud.js:55`

风险等级：中

风险原理与触发条件：演示云函数直接 `console.log("secret: " + secret)`；`secure-network` 打印完整 `event`；前端统一错误提示可能显示后端原始 `message`。如果部署到线上，日志系统会保留 token、用户 ID、业务参数、密钥或环境变量信息。

潜在安全危害：日志平台成为敏感数据泄露源；运维人员或被入侵日志账号可读取 token、密钥、手机号、活动信息；异常消息帮助攻击者识别接口和字段。

可落地修复方案：生产环境删除演示云函数；建立 `safeLogger`，只记录 traceId、action、uid 哈希、错误码，不记录 token、密码、密钥、完整 payload；前端错误统一映射为业务文案，详细堆栈仅写入受控服务端日志；上线前执行 `rg "console\\.log|secret|token"` 审计。

### S-16 举报系统读权限和写入约束不足

问题精准位置：`uniCloud-aliyun/database/surego-reports.schema.json:5`、`uniCloud-aliyun/cloudfunctions/surego-moderation/index.js:135`、`uniCloud-aliyun/cloudfunctions/surego-moderation/index.js:141`、`pages/activity/detail.vue:454`

风险等级：中

风险原理与触发条件：schema 允许任意登录用户读取全部举报；`createReport` 不校验活动是否存在、是否可举报、是否重复举报、举报说明长度和频率；备注直接入库。

潜在安全危害：举报人、被举报活动、举报理由和处理备注泄露；恶意用户可批量刷举报造成运营处理压力；举报内容可被当作恶意文本存储通道。

可落地修复方案：举报表读权限仅限 `admin/operator` 和举报人本人；`createReport` 校验活动存在且非本人创建时的业务规则；同一用户对同一活动同一原因设置 24 小时幂等；增加 IP/用户限流、备注长度限制、敏感词过滤和富文本转义。

### S-17 异常信息和未知 action 直接暴露实现细节

问题精准位置：`common/api/cloud.js:49`、`common/api/cloud.js:55`、`uniCloud-aliyun/cloudfunctions/surego-activity/index.js:295`、`uniCloud-aliyun/cloudfunctions/surego-application/index.js:258`、`uniCloud-aliyun/cloudfunctions/surego-order/index.js:281`

风险等级：低

风险原理与触发条件：前端捕获云函数异常后将 `error.message` 作为 toast 展示；多个云函数返回 `Unsupported action: ${action}`。当 action、字段或数据库错误可控时，攻击者可以通过响应差异枚举接口、action 名称和参数结构。

潜在安全危害：攻击者获得接口字典、错误栈摘要、数据库约束信息，降低后续攻击成本；用户看到内部错误文案影响可信度。

可落地修复方案：所有云函数返回统一结构 `{ code, message: publicMessage, traceId }`；内部错误写服务端日志；未知 action 固定返回 `INVALID_ACTION`；前端仅展示错误码映射后的用户文案。

### S-18 演示云函数和演示页面未隔离生产环境

问题精准位置：`uniCloud-aliyun/cloudfunctions/add/index.js:3`、`uniCloud-aliyun/cloudfunctions/get/index.js:3`、`uniCloud-aliyun/cloudfunctions/update/index.js:3`、`uniCloud-aliyun/cloudfunctions/remove/index.js:3`、`pages/cloudFunction/cloudFunction.vue:49`、`pages/clientDB/permission-demo/permission-demo.vue:122`

风险等级：高

风险原理与触发条件：`add/get/update/remove` 等云函数对 `unicloud-test` 执行未鉴权 CRUD；项目中保留大量 `cloudFunction`、`clientDB`、`validate-demo`、`storage` 演示页面。如果这些路由和云函数随生产包部署，攻击者可直接调用或通过页面触发演示能力。

潜在安全危害：数据库被污染或删除；演示页面暴露 ClientDB 权限模型和云函数调用方式；攻击者利用演示表存储恶意内容或测试权限绕过。

可落地修复方案：生产构建移除所有示例页面、示例 schema、示例云函数和示例 `uni-clientDB-actions`；在 `pages.json` 中按环境条件编译或建立独立 demo 工程；CI 增加阻断规则，检测 `pages/cloudFunction`、`pages/clientDB`、`cloudfunctions/add|get|update|remove` 存在时禁止生产发布。

### S-19 依赖安全治理不足，认证相关依赖锁定旧版本

问题精准位置：`uni_modules/uni-id-pages/uniCloud/cloudfunctions/uni-id-co/package-lock.json:20`、`uni_modules/uni-id-pages/uniCloud/cloudfunctions/uni-id-co/package-lock.json:21`、`uni_modules/uni-id-pages/uniCloud/cloudfunctions/uni-id-co/package-lock.json:106`

风险等级：中

风险原理与触发条件：`uni-id-co` 依赖锁定 `jsonwebtoken@8.5.1` 和 `semver@5.7.2`。GitHub Advisory `GHSA-27h2-hvpr-p74q` 标记 `jsonwebtoken <= 8.5.1` 曾存在 `jwt.verify` 输入校验风险但已于 2023-01-27 withdrawn；即便如此，认证核心依赖停留在旧主版本仍需要评估兼容和修补。GitHub Advisory `GHSA-c2qf-rxjj-qqgw` 显示 `semver@5.7.2` 是 5.x 分支 ReDoS 漏洞的补丁版本，不应误报为当前漏洞，但仍应纳入依赖基线。

潜在安全危害：旧依赖在未来出现新公告时难以及时发现；认证组件兼容层可能缺少新版安全校验；缺少 `npm audit`、Dependabot 或等效供应链扫描会造成漏洞发现滞后。

可落地修复方案：升级 `uni-id-pages/uni-id-co` 到 DCloud 当前稳定版本，重新生成锁文件；在云函数目录执行 `npm audit --production` 或使用 GitHub Dependabot、Snyk、OSS Index；将 `jsonwebtoken` 升级路径评估到 `9.x` 或由新版 `uni-id` 接管；在报告中记录被撤回公告，避免错误定级，但保留旧版本升级任务。

参考来源：`https://github.com/advisories/GHSA-27h2-hvpr-p74q`、`https://github.com/advisories/GHSA-c2qf-rxjj-qqgw`

### S-20 App 路由和分享入口缺少统一 URL 安全策略

问题精准位置：`common/utils/route.js:154`、`common/utils/route.js:191`、`common/utils/route.js:334`、`pages/auth/login.vue:122`、`pages/share/poster.vue:148`

风险等级：低

风险原理与触发条件：项目存在多处根据参数跳转、登录后重定向、复制分享路径的逻辑。虽然部分工具函数存在 `safeFallbackUrl`，但整体缺少统一的 route allowlist、参数 schema 和分享路径签名策略。

潜在安全危害：构造异常 path 可造成错误页面、绕过预期流程或诱导用户进入敏感页面；分享路径被篡改后可能引导至 WebView、登录或非预期活动详情。

可落地修复方案：建立 `routeAllowlist`，所有 `navigateTo/redirectTo/reLaunch` 只允许项目内注册路径；登录 redirect 只接受 allowlist 内相对路径；分享链接只包含业务 ID，不包含任意 URL；对活动 ID、用户 ID 等路由参数做格式校验。

## 三、优先级修复路线

第一优先级：下线或加固 `user-center`，统一 SureGo 云函数鉴权，关闭生产演示表和演示云函数，修复订单支付状态和报名状态的客户端信任问题。

第二优先级：收紧 schema 权限，重写签到凭证，修复用户资料泄露，建立消息创建白名单，删除高危 App 权限。

第三优先级：补齐上传校验、WebView 白名单、日志脱敏、异常统一响应、依赖安全扫描和路由 allowlist。
